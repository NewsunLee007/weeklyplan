# 修复用户列表分页与 AI 知识库回复问题的计划

## 1. 现状分析 (Current State Analysis)
根据您提供的终端日志与报错信息，经过深入排查，发现以下两个关键问题：

### 问题 1：用户列表为何只显示 10 条，且总数为 0？
虽然上次我们修复了 `LIMIT` 和 `OFFSET` 的参数错位问题，但这里隐藏着一个更底层的 JavaScript 语法陷阱：
在 `users.js`（以及 `plans.js`）中，获取总数的代码为：
```javascript
const total = await queryOne(...)?.cnt || 0;
```
由于 JavaScript 中**可选链操作符 `?.` 的优先级高于 `await`**，这句代码实际上会被解析为 `await (queryOne(...)?.cnt)`。因为 `queryOne` 返回的是一个 `Promise` 对象，而 `Promise` 没有 `.cnt` 属性，所以 `?.cnt` 直接计算为 `undefined`，进而导致 `total` 永远等于 `0`！
这就是为什么您的 15 个用户已经成功存入数据库（所以没有报错），但前端分页组件因为接收到 `total = 0`，认为没有更多数据，从而截断了第 2 页，导致您在界面上看不到新导入的用户。

### 问题 2：AI 为什么说自己没有调用知识库？
从您的截图可以看到，AI 回复：“在之前的对话中，您提到了'新纪元工作内容大纲'……”。这实际上**证明了系统已经成功将知识库内容提取并发送给了 AI**（否则它不可能知道这个大纲）。
**那为什么它说自己没调用呢？** 这是大语言模型（LLM）的常见认知错觉。因为系统在底层是直接将知识库文本拼接到了 `System Prompt` 中发送给它的，AI 并没有“主动点击按钮去查询”的动作，所以它会误以为这些内容是“您在聊天上下文中发给它的”。当您质问它时，它触发了安全限制，声明自己“无法直接访问本地数据库”。

## 2. 拟议修改 (Proposed Changes)

### 修复 1：修正所有分页接口的 `total` 计数逻辑
**涉及文件**：
- `api/_lib/routes/users.js` 和 `backend/src/routes/users.js`
- `api/_lib/routes/plans.js` 和 `backend/src/routes/plans.js`

**修改方案**：
将所有的链式调用拆开，确保 `await` 优先执行并获取到结果对象：
```javascript
const totalRes = await queryOne(...);
const total = totalRes?.cnt ? parseInt(totalRes.cnt, 10) : 0;
```
这样修改后，PostgreSQL 正确返回的 `{ cnt: '25' }` 就能被正确解析为 `25`，前端分页组件恢复正常，第二页的新用户就会显示出来。

### 修复 2：优化 AI 智能助手的系统提示词 (Prompt Engineering)
**涉及文件**：
- `api/_lib/routes/ai.js` 和 `backend/src/routes/ai.js`

**修改方案**：
修改 `systemPrompt`，向 AI 明确注入其当前的角色和能力设定。
将原本的：
`'你是一个智能助手，帮助用户进行工作计划管理、分析和建议。请基于提供的知识库信息来回答用户的问题。'`
改为：
`'你是一个内置于系统中的智能助手，已经成功连接并调用了用户的系统知识库。以下【知识库信息】中的内容就是你从数据库中实时检索到的最新业务知识。如果用户问你是否调用了知识库，请明确且自信地回答你已经成功读取并正在使用知识库中的内容，切勿声称自己无法访问数据库。'`
这样可以完美解决 AI 的“身份认知错乱”问题，让您的知识库真正具备“专属 AI”的体验。
