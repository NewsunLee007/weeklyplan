# 优化审核流程与增加二次修改、导出功能的计划

## 1. 现状分析 (Current State Analysis)
目前系统的审核流程是固定的三步：提交 -> 部门审核 -> 办公室审核 -> 校长审核发布。但在实际校园管理中，存在以下特殊情况和需求没有被满足：
1. **部门主任自己提交的计划**：目前仍然需要他们自己去点“部门审核通过”，略显繁琐。
2. **办公室主任的终审权与导出需求**：办公室主任应该具备“一键发布”的最终审核权，同时可以在审核时将全校计划导出为 Word 文档（可能用于纸质审批或存档）。
3. **二次修改需求**：一旦计划发布（状态变为 `PUBLISHED`），目前前端界面会将其锁定为只读。但实际工作中，可能存在临时调整，需要办公室主任具备对已发布计划的“二次修改”权限。

## 2. 拟议修改 (Proposed Changes)

为了满足以上需求，我计划在前后端进行如下调整：

### 2.1 后端 API 调整
1. **部门主任免审机制 (`routes/plans.js`)**
   - 修改新建计划 (`POST /`) 接口：如果当前登录用户的角色是 `DEPT_HEAD`（部门主任），则新创建的计划初始状态直接设为 `DEPT_APPROVED`（待办公室主任审核），跳过第一步。
2. **办公室主任终审与一键发布 (`routes/reviews.js`)**
   - 修改待审核列表 (`GET /pending` & `GET /consolidated`)：允许办公室主任查看到 `DEPT_APPROVED` 和 `OFFICE_APPROVED` 状态的计划。
   - 修改审核通过接口 (`POST /:planId/approve` & 批量接口)：
     - 增加 `publish` 参数。如果办公室主任调用接口时传入 `publish: true`，则跳过校长审核步骤，直接将状态置为 `PUBLISHED`（一键发布）。
3. **已发布计划二次修改 (`routes/plans.js`)**
   - 新增 `PUT /:id/published-items` 接口，专门允许 `OFFICE_HEAD` 和 `ADMIN` 对状态为 `PUBLISHED` 的计划条目进行增删改。
4. **Word 导出兼容审核中状态 (`routes/export.js`)**
   - 修改全校周计划汇总导出接口 (`GET /weekly-summary/:weekNumber`)，增加 `status` 参数，允许办公室主任在审核阶段（不仅是发布后）就能导出待发布的整合 Word 文档，以供纸质审批。

### 2.2 前端界面调整
1. **单计划审核详情页 (`views/review/ReviewDetail.vue`)**
   - 如果是办公室主任且计划处于待审核状态，底部显示两个按钮：**【通过并交校长审核】**和**【一键发布】**。
   - 如果计划已发布且当前用户是办公室主任，解开条目的只读锁定，并在底部显示**【保存二次修改】**按钮，调用新的二次修改接口。
   - 在页面右上角增加**【导出为 Word】**按钮。
2. **全校整合审核页 (`views/review/ReviewConsolidated.vue`)**
   - 同样增加**【批量通过并交校长审核】**和**【批量一键发布】**两个按钮供办公室主任选择。
   - 页面顶部增加**【导出全校整合 Word】**按钮，方便打印给校长签字。

## 3. 假设与决策 (Assumptions & Decisions)
- **决策**：对于部门主任自己提交的计划，直接进入“待办公室主任审核”状态。这简化了部门主任的工作流。
- **决策**：办公室主任可以在系统内直接一键发布，也可以点击“交校长审核”并在系统外拿着导出的 Word 给校长签字，签完字后再在系统中点击“一键发布”。这赋予了流程最大的灵活性。
- **决策**：二次修改的权限仅开放给“办公室主任”和“系统管理员”，普通教师和部门主任在发布后无法再修改。

## 4. 验证步骤 (Verification Steps)
1. 实施计划代码并进行全量编译（`npm run build`）。
2. 使用“部门主任”账号新建计划，验证是否直接进入办公室主任的待办列表。
3. 使用“办公室主任”账号进入审核页，验证是否出现【一键发布】和【导出Word】按钮。
4. 验证办公室主任能否在“已发布”的计划详情中对条目进行二次修改和保存。
