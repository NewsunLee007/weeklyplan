# AI 助手全场景上下文打通计划

## 1. 现状分析 (Current State Analysis)
根据用户的需求和对系统源码的探索：
- **当前 AI 聊天的现状**：系统中的 AI 对话接口（`/api/ai/chat`）虽然已经接入了系统配置的 API Key 和大模型，并且成功打通了**系统知识库**（`getKnowledgeContext`），但它**缺失了对用户身份的深度认知**，也**没有读取本周的工作计划数据**。
- **用户的痛点**：AI 不知道现在和它聊天的是校长、教务处主任还是普通老师；AI 也不知道这周学校和各个部门到底安排了哪些工作。导致 AI 的回答往往是通用的废话，无法结合学校实际情况给出有价值的建议。

## 2. 拟议修改 (Proposed Changes)

为了让 AI 真正成为懂学校、懂老师的智能助手，我们需要在后端的聊天接口中注入三大核心上下文：

### 2.1 注入“用户身份感知”上下文
- **文件**：`api/_lib/routes/ai.js` 和 `backend/src/routes/ai.js`
- **修改**：利用中间件提供的 `req.user`（包含 `real_name`, `role`, `department_name` 等），在构建 `systemPrompt` 时，明确告诉 AI 当前对话者的真实姓名、所属部门以及系统职务（如校长、系统管理员、普通教师等）。让 AI 的回答语气和视角因人而异。

### 2.2 注入“当前周次全校/部门计划”上下文
- **文件**：`api/_lib/routes/ai.js` 和 `backend/src/routes/ai.js`
- **修改**：
  1. 新增一个 `getCurrentWeekPlans(user)` 函数。
  2. 首先通过查询 `sys_config` 获取 `current_semester`（当前学期）和 `current_week_start`（开学日期），动态计算出**系统当前的真实周次**。
  3. 查询数据库 `biz_week_plan` 和 `biz_plan_item`，拉取当前周次下**所有已发布（PUBLISHED）的全校计划**。
  4. 如果当前用户有自己草稿或待审核的计划，也一并拉取出来作为专属上下文。
  5. 将这些计划按部门、星期、工作内容拼接成易于大模型阅读的文本（如 `【教务处 - 周二】：完成期中考试排表`），追加到 `systemPrompt` 中。

### 2.3 优化 System Prompt 指令
- **修改**：在系统提示词中加入强指令：“你已接入学校的真实数据库。请仔细阅读下方的【系统知识库】和【本周工作计划行事历】。当用户询问工作安排、学校制度或寻求建议时，**必须**优先从这些上下文中提取真实信息进行回答，切勿编造。”

## 3. 验证步骤 (Verification Steps)
1. 以不同角色的账户（如普通老师、校长）登录系统，打开右下角的 AI 智能助手。
2. 提问：“我是谁？我在哪个部门？”验证 AI 是否能准确报出您的姓名和部门。
3. 提问：“这周学校有什么工作安排？”或者“我们部门这周要干嘛？”，验证 AI 是否能精准读取并总结出刚才系统中已发布的周计划内容。
